# Introduction

![Status: Planned](https://img.shields.io/badge/status-Planned-blue)

Plan này mô tả phase `04` cho việc mở rộng `control_api` và `control_messaging` để hỗ trợ add/remove camera runtime trong `vms-engine` khi pipeline đang chạy. Scope của phase này là manual source mode `sources.type: nvurisrcbin`, tái sử dụng `PipelineManager`, `RuntimeStreamManager`, `RuntimeControlHandler`, và `RuntimeControlMessageConsumer` hiện có, đồng thời bổ sung contract lỗi machine-readable để Python backend map message ổn định cho client. Mỗi lần topology update thành công hoặc thất bại sau khi đã mutate pipeline state phải dump file `.dot` nếu `pipeline.dot_file_dir` được cấu hình, và tài liệu phải phản ánh rõ custom engine control path này khác với DeepStream embedded REST của `nvmultiurisrcbin`.

## 1. Requirements & Constraints

- **REQ-001**: Runtime camera control chỉ áp dụng cho manual source mode `sources.type = "nvurisrcbin"` trong [core/include/engine/core/config/config_types.hpp](/home/vms/Lantana/Dev/vms-engine/core/include/engine/core/config/config_types.hpp). Nếu pipeline dùng `nvmultiurisrcbin`, control path mới phải trả lỗi machine-readable `SRCCTL_UNSUPPORTED_SOURCE_MODE` thay vì âm thầm fallback sang DeepStream embedded REST.
- **REQ-002**: HTTP API phải dùng chung `RuntimeControlHandler` hiện có trong [infrastructure/control/include/engine/infrastructure/control/runtime_control_handler.hpp](/home/vms/Lantana/Dev/vms-engine/infrastructure/control/include/engine/infrastructure/control/runtime_control_handler.hpp) và [infrastructure/control/src/runtime_control_handler.cpp](/home/vms/Lantana/Dev/vms-engine/infrastructure/control/src/runtime_control_handler.cpp); không được tạo code path camera control riêng chỉ tồn tại trong REST adapter.
- **REQ-003**: Broker command qua `control_messaging` phải dùng cùng handler và cùng response schema với HTTP API. Các response qua HTTP và broker phải có cùng `status_code`, `ok`, `error`, `error_code`, `message`, `pipeline_id`, `camera_id`, và metadata `dot_file` khi có.
- **REQ-004**: Bổ sung camera-control contract typed ở lớp `core/` trước khi implement chi tiết ở `pipeline/`. `bool IPipelineManager::add_source(...)` và `bool IPipelineManager::remove_source(...)` hiện tại không đủ để truyền lý do lỗi cho Python backend.
- **REQ-005**: Contract typed mới phải định nghĩa tối thiểu `RuntimeSourceInfo`, `RuntimeSourceMutationRequest`, `RuntimeSourceMutationResult`, và `RuntimeSourceErrorCode` trong `core/`. `RuntimeSourceMutationResult` phải chứa tối thiểu các field `success`, `http_status`, `error_code`, `message`, `camera_id`, `source_index`, `active_source_count`, và `dot_file_path`.
- **REQ-006**: Bộ mã lỗi custom phải ổn định, deterministic, và dùng được trực tiếp từ Python. Ít nhất phải có các mã sau: `SRCCTL_INVALID_REQUEST`, `SRCCTL_PIPELINE_NOT_FOUND`, `SRCCTL_UNSUPPORTED_SOURCE_MODE`, `SRCCTL_DUPLICATE_CAMERA_ID`, `SRCCTL_CAMERA_NOT_FOUND`, `SRCCTL_MAX_SOURCES_REACHED`, `SRCCTL_BUILD_SOURCE_FAILED`, `SRCCTL_REQUEST_PAD_FAILED`, `SRCCTL_LINK_SOURCE_FAILED`, `SRCCTL_OPERATION_TIMEOUT`, `SRCCTL_DOT_DUMP_FAILED`, `SRCCTL_INTERNAL_ERROR`.
- **REQ-007**: API HTTP phải expose tối thiểu các route mới sau trong [infrastructure/rest_api/src/pistache_server.cpp](/home/vms/Lantana/Dev/vms-engine/infrastructure/rest_api/src/pistache_server.cpp): `GET /api/v1/pipelines/{pipeline_id}/sources`, `POST /api/v1/pipelines/{pipeline_id}/sources`, `DELETE /api/v1/pipelines/{pipeline_id}/sources/{camera_id}`.
- **REQ-008**: Message contract mới cho `control_messaging` phải expose tối thiểu các `type`: `list_sources`, `add_source`, `remove_source`. Payload của `add_source` phải chứa object `camera` với `id` và `uri`. Payload của `remove_source` phải chứa `camera_id`.
- **REQ-009**: Runtime control phải xử lý đúng trường hợp pipeline khởi tạo với sẵn camera trong `sources.cameras`. Các camera seed ban đầu phải xuất hiện trong source list, remove được qua control path mới, và add lại được cùng `camera_id` sau khi remove thành công.
- **REQ-010**: Không được cho phép thêm trùng `camera_id` với bất kỳ source nào đang active, bao gồm cả camera seed ban đầu và camera đã add runtime. Lỗi duplicate phải map thành HTTP `409 Conflict` với `error_code = SRCCTL_DUPLICATE_CAMERA_ID`.
- **REQ-011**: Sau khi remove thành công một camera, cùng `camera_id` phải add lại được mà không cần restart pipeline. `RuntimeStreamManager` phải release đủ state nội bộ để reuse source slot hoặc cấp source slot mới hợp lệ.
- **REQ-012**: Mỗi lần add/remove camera qua control API hoặc control messaging phải tạo DOT snapshot nếu `config.pipeline.dot_file_dir` không rỗng. Tên file phải có prefix theo operation và `camera_id`, ví dụ `runtime_add_camera_01` hoặc `runtime_remove_camera_01`, và phải được trả lại trong response khi dump thành công.
- **REQ-013**: Nếu mutate pipeline thành công nhưng dump DOT thất bại, API không được rollback runtime mutation. Response phải trả `ok=true` nhưng kèm warning machine-readable hoặc metadata rõ ràng rằng dot dump thất bại. Nếu team muốn strict failure thay vì warning thì phải ghi rõ decision trong doc trước khi code; phase này mặc định chọn non-fatal DOT failure.
- **REQ-014**: `RuntimeStreamManager` phải trả được lỗi chi tiết thay vì chỉ `true/false` cho các failure path trong [pipeline/src/runtime_stream_manager.cpp](/home/vms/Lantana/Dev/vms-engine/pipeline/src/runtime_stream_manager.cpp), gồm các tình huống request pad thất bại, build source thất bại, resolve raw source element thất bại, link source pad thất bại, và not found khi remove.
- **REQ-015**: `RuntimeControlHandler` phải map các lỗi typed từ `PipelineManager` sang HTTP status code ổn định. Mapping mặc định: `400` cho invalid request, `404` cho pipeline/camera not found, `409` cho duplicate hoặc conflict, `422` cho unsupported mode nếu request hợp lệ nhưng pipeline không hỗ trợ, `507` cho capacity reached nếu vượt `max_sources`, `500` cho internal/runtime mutation failure.
- **REQ-016**: `PistacheServer` phải tiếp tục đóng connection sau mỗi response path như note trong memory `vms-engine-control-api-connection-close-2026-03-11.md`, vì add/remove là mutation request và client Python không được treo chờ response dù side effect đã hoàn tất.
- **REQ-017**: `RuntimeControlMessageConsumer` trong [infrastructure/control/src/runtime_control_message_consumer.cpp](/home/vms/Lantana/Dev/vms-engine/infrastructure/control/src/runtime_control_message_consumer.cpp) phải publish reply cho `list_sources`, `add_source`, `remove_source` với cùng `error_code` và `message` như HTTP. Reply không được chỉ có `ok=true/false`.
- **REQ-018**: `RuntimeControlHandler` phải có validation request riêng cho camera payload. `camera.id` và `camera.uri` là mandatory; `camera.id` không được rỗng; `camera.uri` không được rỗng; mọi field dư phải bị ignore hoặc preserved trong payload echo nhưng không được làm handler crash.
- **REQ-019**: Source list response phải trả đủ context cho Python debug và reconciliation, tối thiểu gồm `camera_id`, `source_index`, `state`, `is_seeded`, và `uri` nếu runtime manager còn biết giá trị đó. Nếu không giữ được `uri`, doc phải ghi rõ limitation này trước khi triển khai.
- **REQ-020**: Add/remove runtime phải tiếp tục tuân thủ các guard đã được ghi nhận trong memory `vms-engine-runtime-remove-state-transition-2026-03-11.md` và `vms-engine-runtime-remove-needs-keyframe-2026-03-11.md`: add/remove là mutually exclusive, remove phải chờ source về `GST_STATE_NULL` trước khi bỏ khỏi bin, và sau topology change phải trigger output recovery path hoặc keyframe request nếu repo hiện đang phụ thuộc logic đó.
- **REQ-021**: Plan này phải bao phủ cả error-path test cho các trường hợp: invalid JSON, unsupported message type, pipeline id sai, remove camera không tồn tại, add camera trùng id, add camera thiếu `uri`, add vượt `max_sources`, và request vào pipeline không phải manual source mode.
- **REQ-022**: Tài liệu ở [docs/architecture/deepstream/11_runtime_element_control.md](/home/vms/Lantana/Dev/vms-engine/docs/architecture/deepstream/11_runtime_element_control.md) phải được mở rộng để bao phủ runtime camera control qua custom engine transport. ## 3. Alternatives

- **ALT-001**: Dùng luôn DeepStream embedded REST cho mọi mode source. Không chọn vì manual mode `sources.type = nvurisrcbin` hiện đi qua `RuntimeStreamManager`, không có embedded REST tương đương, và user đang yêu cầu control path qua `control_api` cùng `control_messaging` của engine.
- **ALT-002**: Giữ `bool add_source/remove_source` rồi ánh xạ mọi lỗi về `400` hoặc `500`. Không chọn vì Python backend cần mã lỗi ổn định để hiển thị message đúng cho client và phân biệt duplicate, not found, capacity, unsupported mode, timeout.
- **ALT-003**: Tạo REST route riêng thao tác trực tiếp với `RuntimeStreamManager` và để broker path dùng logic khác. Không chọn vì điều này phá vỡ pattern transport-agnostic đã được dùng cho runtime property control.
- **ALT-004**: Tạo DOT dump như một endpoint hoặc command riêng sau khi add/remove xong. Không chọn vì user yêu cầu dump sau mỗi update và operator không nên phải gọi thêm bước thứ hai.
- **ALT-005**: Fail toàn bộ request nếu DOT dump thất bại. Không chọn cho mặc định phase này vì topology mutation đã thành công, rollback là rủi ro hơn; hợp lý hơn là trả success kèm warning machine-readable.
- **ALT-006**: Cho phép add trùng `camera_id` nếu URI khác. Không chọn vì `camera_id` là identity key ở cả config, runtime manager, downstream reconciliation, và remove path.

## 7. Risks & Assumptions

- **RISK-001**: `IPipelineManager` signature change có thể kéo theo nhiều caller trong app/infrastructure; nếu không giữ wrapper compatibility thì refactor scope sẽ lan rộng hơn dự kiến.
- **RISK-002**: DOT dump path của GStreamer phụ thuộc environment và quyền ghi thư mục. Nếu helper không set đúng output directory thì operator sẽ nghĩ feature không hoạt động dù runtime mutation thành công.
- **RISK-003**: Add/remove runtime hiện có guard timing nhạy cảm với state transition; nếu typed-result refactor vô tình bỏ guard chờ state hoặc keyframe recovery thì RTSP/WebRTC output có thể regression dù API response nhìn vẫn thành công.
- **RISK-004**: Response schema mới cho broker command sẽ cần downstream Python update đồng bộ để đọc `error_code` và `message`; nếu backend vẫn parse kiểu cũ chỉ dựa `ok`, lợi ích typed errors sẽ chưa được khai thác hết.
- **RISK-005**: `source list` muốn trả `uri` cho seed sources và runtime sources đòi hỏi runtime manager lưu metadata nhất quán; hiện file [pipeline/src/runtime_stream_manager.cpp](/home/vms/Lantana/Dev/vms-engine/pipeline/src/runtime_stream_manager.cpp) chưa giữ field này trong `StreamSlot`.
- **ASSUMPTION-001**: Runtime add/remove target của phase này là manual source mode đang được sample config dev sử dụng, không phải legacy DeepStream REST path.
- **ASSUMPTION-002**: `pipeline.dot_file_dir` sẽ được bật trong dev config và container có quyền ghi vào thư mục đó.
- **ASSUMPTION-003**: Python backend cần machine-readable codes trước hết cho API layer; wording user-facing message sẽ được map phía Python chứ không hardcode toàn bộ trong engine.
